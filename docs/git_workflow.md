# Git Workflow

## Main Branch

The `main` branch stores stable and reviewed code only. Team members should not directly push large unreviewed changes to `main`.

## Feature Branches

Each team member should create a separate branch for their assigned module.

Recommended branch name:  

- `Bowen/phase2-unsupervised`
- `member2/phase3-supervised`

## Commit Guidelines

Commits should be small, frequent, and meaningful.
Avoid vague messages.


## Pull Request Workflow

1. Create a feature branch.
2. Make small commits.
3. Push the branch to GitHub.
4. Open a pull request.
5. Ask at least one teammate to review.
6. Merge only after the code is understandable and does not break the project.

## Contribution Visibility

If two members work together on the same file, use separate commits or co-authored commits.

## GitHub Issues

GitHub Issues should be used to track tasks, bugs, questions, and project decisions.

Recommended issue types:

- `task`: planned project work
- `bug`: code error or unexpected result
- `question`: uncertainty about dataset, method, or implementation
- `documentation`: README, report, or documentation updates
- `experiment`: model or feature engineering experiment
